# Codex in-turn steering

This document records the implementation boundary for adding non-interrupting follow-up messages to
Paseo's Codex app-server provider. Research was performed against the official Codex app-server
manual and schemas generated by `codex-cli 0.144.6`.

## Terminology and upstream behavior

The upstream name is **steer**. The stable app-server request is `turn/steer`. It appends user input
to the currently active turn rather than interrupting it or creating another turn.

The generated request shape is:

```ts
interface TurnSteerParams {
  threadId: string;
  clientUserMessageId?: string | null;
  input: UserInput[];
  expectedTurnId: string;
}
```

The response contains the active `turnId`. `expectedTurnId` is a required concurrency precondition;
app-server rejects a stale request rather than steering a newer turn by accident. Review and compact
turns can return `activeTurnNotSteerable` with kind `review` or `compact`.

Steering is delivered at an app-server/model-loop safe point without canceling the model's current
work. It is appropriate to describe this as “the active turn sees the new instruction after its
current work reaches the next inference boundary.” Do not promise that the instruction is literally
attached to the next tool result; that timing is an implementation detail rather than an app-server
protocol guarantee.

There is no stable app-server RPC for a durable “run this after the current turn” queue in the
generated schema. Codex clients implement that UI behavior above app-server. The bounded ingress
queue documented for app-server's experimental WebSocket transport is only overload control and is
not a user-message follow-up queue.

## Current Paseo behavior

Paseo currently exposes two running-agent choices:

- **Queue** stores the message in the app session store. `SessionContext` sends it when the agent
  becomes idle. Other clients and the daemon do not own or observe that queue.
- **Send and interrupt** reaches `sendPromptToAgent()`, which calls `replaceAgentRun()` for an
  in-flight run. The manager invokes provider `interrupt()` and starts a new turn.

`CodexAppServerAgentSession` already tracks both the thread id and native turn id and implements
`turn/start` and `turn/interrupt`, so `turn/steer` fits without replacing the provider transport.

## Recommended implementation boundary

Implement steering as an explicit delivery mode on the existing send operation:

1. Add optional steering support to `AgentSession` and a `supportsSteering` provider capability.
2. Add `AgentManager.steerAgent()`. It must require an active foreground run, call the current
   session's steering method, and leave run ownership, lifecycle, waiters, and terminal-event handling
   unchanged.
3. In the Codex session, build the same `UserInput[]` used by `turn/start` and call `turn/steer` with
   `currentThreadId` and the native active turn id. Parse the `turn/start` response and retain its turn
   id immediately, rather than relying only on the later `turn/started` notification.
4. Add optional `delivery: "steer" | "interrupt"` to `send_agent_message_request`. Omission retains
   the historical interrupt-and-replace behavior, so official clients keep working with the fork
   daemon. A fork client sends `delivery: "steer"` only after the daemon advertises
   `server_info.features.agentTurnSteer` and the active agent advertises `supportsSteering`; it omits
   the field for an official daemon.
5. While an agent is running, show a wrench and **Steer current turn** only when both capabilities
   are present. Otherwise retain the arrow and **Send and interrupt**. Queue and the explicit
   send-now action on a queued item remain distinct; send-now deliberately interrupts.

The existing response reports acceptance or a user-facing error. On a rejected steer the client
restores the draft; it must not silently retry the same message as an interrupt because that can
duplicate an accepted steer whose response was lost.

## Timeline and concurrency invariants

- A successful steer belongs to the existing turn. It must not emit another Paseo `turn_started`,
  replace the foreground run token, or create a second stream drainer.
- Prefer `clientUserMessageId` for end-to-end deduplication. Confirm through the fake app-server and a
  real-provider test that app-server emits the steered user item once and that Paseo persists it once.
- The native turn id from `turn/start` response and `turn/started` notification must agree. A mismatch
  is a provider protocol error, not a reason to steer whichever turn happens to be active.
- Multiple clients may submit steering concurrently. Each request carries the turn id the daemon
  observed; app-server's required precondition prevents a late request from crossing a turn boundary.
- Slash commands and output-schema-changing prompts need an explicit policy. The first implementation
  should steer ordinary text, images, attachments, skills, and mentions, while keeping Paseo
  out-of-band commands on their existing path.

## Test slices

1. Provider unit test: `turn/steer` receives the converted input and exact active turn id without a
   preceding `turn/interrupt`.
2. Provider race tests: reject before a turn id exists, reject a mismatched response id, and surface
   `activeTurnNotSteerable`.
3. Manager test: steering does not allocate, cancel, or settle a run.
4. Protocol/client test: an old daemon has no steering feature and the UI does not offer it.
5. Real Codex test: steer while a tool is running and confirm the instruction appears in the same
   native turn and same Paseo timeline epoch.
